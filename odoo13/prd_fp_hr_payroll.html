Product Requirements Document

FP HR Payroll Ecosystem (Odoo 13 Custom Implementation)

Ecosystem Modules Overview

Modul Utama: fp_hr_payroll Terkait: if_hr_payroll_attendance Terkait: if_hr_payroll_journal Terkait: if_loan Terkait: if_mobile_payroll

1. Ringkasan Dokumen

Dokumen ini mendefinisikan Product Requirements Document (PRD) untuk ekosistem payroll custom Odoo 13 yang berpusat pada modul fp_hr_payroll. PRD ini disusun berdasarkan implementasi aktual pada codebase, bukan asumsi fungsional generik. Fokusnya adalah menjelaskan kebutuhan bisnis, ruang lingkup, aktor, data model utama, alur end-to-end, integrasi antar modul, aturan bisnis, risiko, dan kriteria penerimaan.

Tujuan utamanya adalah menyediakan payroll engine berbasis Odoo yang:

  • Mengelola master payroll dan regulasi pemerintah.
  • Menyinkronkan data master dan transaksi ke backend payroll eksternal.
  • Menerima hasil kalkulasi payroll dari backend tersebut.
  • Menyajikan, memfinalisasi, melaporkan, dan menjurnal hasil payroll di Odoo.
  • Mendukung payroll regular dan off-cycle.
  • Mendukung attendance-driven payroll, loan deduction, payroll journal, dan mobile payslip.

2. Latar Belakang

Payroll pada implementasi ini tidak dihitung penuh oleh Odoo. Odoo bertindak sebagai:

  • Sumber master data karyawan, kontrak, company, cutoff, dan konfigurasi.
  • Antarmuka operasional HR dan payroll.
  • Penyimpan hasil kalkulasi payroll.
  • Generator laporan dan dokumen.
  • Orchestrator finalisasi dan jurnal.

Kalkulasi inti payroll, pajak, BPJS, TER, THR, pesangon, dan sebagian report tahunan bergantung pada backend eksternal yang diakses melalui model hr.payroll.api.config.

Dengan demikian, kebutuhan bisnis yang diakomodasi modul ini adalah:

  • Payroll Indonesia dengan aturan BPJS, PPh21, PTKP, TER, THR, dan pesangon.
  • Multi-company.
  • Payroll regular dan off-cycle.
  • Dukungan attendance-based payroll untuk daily worker dan overtime.
  • Kebutuhan banking export, tax report, BPJS report, dan jurnal akuntansi.
  • Kemampuan koreksi atas payroll yang telah finalized dalam kondisi tertentu.

3. Tujuan Produk

3.1 Tujuan Bisnis

  • Memastikan proses payroll bulanan dan off-cycle berjalan konsisten lintas entitas perusahaan.
  • Mengurangi perhitungan manual payroll dan risiko salah hitung.
  • Menjaga auditability atas konfigurasi payroll dan perubahan regulasi.
  • Menyediakan output payroll yang siap pakai untuk HR, Finance, Accounting, dan kebutuhan pelaporan pajak/BPJS.
  • Menghubungkan payroll ke proses akuntansi dan mobile distribution.

3.2 Tujuan Operasional

  • Menyediakan satu sumber kendali payroll di Odoo.
  • Menjaga sinkronisasi data master ke backend payroll.
  • Menghasilkan payslip draft otomatis untuk batch regular.
  • Mendukung finalisasi terkontrol dengan validasi period backend.
  • Mendukung recalculation saat master/payroll component berubah.
  • Menyediakan report statutory dan operational.

3.3 Non-Goal

  • Tidak menggantikan backend kalkulasi eksternal.
  • Tidak mendefinisikan formula payroll murni di Odoo salary rule standar.
  • Tidak menutup semua use case HR; fokus di payroll-related flow.

4. Scope

4.1 In Scope

  • Konfigurasi API payroll backend.
  • Sinkronisasi master company, government regulation, employee, contract.
  • Payroll period management / Payroll initialization / accumulated payroll.
  • Regular payroll batch, THR batch, Pesangon batch, Other off-cycle batch.
  • Recalculation salary component, Correction flow.
  • Tax report 1721, BPJS / BPJSTK reporting, E-banking export, Payroll batch reporting.
  • Attendance-based payroll extension, Payroll journal accounting extension, Loan deduction extension, Mobile payslip attachment extension.

4.2 Out of Scope

  • Desain backend payroll API.
  • Infrastruktur hosting API eksternal.
  • Formula salary rule Odoo native sebagai engine utama.
  • ESS portal penuh untuk payroll selain file mobile attachment.

5.4 User Persona / Aktor

  • 5.4.1 Payroll Officer: Menjalankan batch payroll, memeriksa payslip draft, mengupdate salary components tertentu, menghasilkan report dan export bank, submit batch.
  • 5.4.2 Payroll Manager: Mengelola konfigurasi payroll, memfinalisasi batch, mengelola government regulation, menjalankan correction bila diizinkan, menilai consistency payroll output.
  • 5.4.3 HR Officer / HR Manager: Mengelola employee master, mengelola contract, memastikan field payroll/tax employee valid, mengelola THR/pesangon input tertentu.
  • 5.4.4 Attendance Admin / Supervisor: Menyetujui attendance, overtime, leave. Memastikan data attendance complete sebelum payroll run.
  • 5.4.5 Accounting / Finance: Menggunakan payroll journal, menggunakan export bank, menggunakan report total payroll, liabilities, PPh21, BPJS.
  • 5.4.6 Employee / Mobile Consumer: Menerima payslip attachment untuk konsumsi mobile.
  • 5.4.7 System Administrator: Mengisi API host, melakukan sync master, troubleshooting mismatch period / sync issue, upgrade module.

6. Problem Statement

Organisasi membutuhkan payroll Indonesia yang:

  • Menggabungkan data HR, kontrak, dan attendance.
  • Menghitung komponen salary, pajak, dan BPJS secara benar.
  • Mendukung skenario regular dan off-cycle.
  • Menghasilkan dokumen statutory dan operasional.
  • Terhubung ke accounting.
  • Dapat diaudit dan disesuaikan per company.

Tantangan utama: Banyak aturan lokal payroll dan tax. Payroll sangat sensitif terhadap cutoff, resign, join date, backpay, dan daily worker. Perubahan employee/contract harus tercermin ke draft payroll tanpa merusak data finalized. Harus ada integrasi yang aman ke backend payroll eksternal. Finance membutuhkan jurnal dan export bank yang konsisten dengan hasil payroll.

7. Solusi Produk

Produk ini menggunakan Odoo sebagai control center dan backend payroll eksternal sebagai compute engine. Solusinya terdiri dari 6 pilar:

  1. Master sync: Company, regulation, employee, contract dikirim ke backend.
  2. Batch orchestration: Odoo membuat batch, mengambil hasil dari backend, lalu membuat payslip lokal.
  3. User adjustment: User dapat mengubah komponen tertentu di payslip draft dan meminta recalculation.
  4. Finalization: Batch difinalisasi di Odoo lalu dipost ke backend.
  5. Reporting and export: Odoo menyiapkan payroll report, tax report, BPJS report, dan bank export.
  6. Extensions: Attendance, journal, loan, mobile berjalan sebagai layer di atas payroll inti.

8. Arsitektur Produk (Diperbarui)

8.1 Konsep Arsitektur

  • Odoo menyimpan master dan transaksi.
  • Odoo memanggil REST API payroll backend.
  • Backend mengembalikan hasil per contract/employee.
  • Odoo memetakan hasil ke hr.payslip.
  • Finalization dilakukan dua arah: status di Odoo dan posting final ke backend.
  • Modul extension membaca hasil hr.payslip untuk proses lanjut.

9. Data Domain & Extension Modules

9.1 Requirement Fungsional

  • 9.1.1 API Configuration: Menyimpan 1 host aktif, action sync all, disable_sync.
  • 9.1.2 Company Sync: Memicu recalculation jika config berubah.
  • 9.1.3 Government Regulation: Scheduler, history change.
  • 9.1.4 Employee & Contract Master: Memicu update draft payroll jika relevan. Mencegah delete jika payslip ada.
  • 9.1.5 Payroll Run: Regular & Off-Cycle (THR, Pesangon, Others). Memastikan period cocok, hitung hasil.
  • 9.1.6 Payslip Adjustment: Tambah/ubah irregular allowance, overwrite BPJS, recalculation trigger.
  • 9.1.7 Recalculation: Menghitung ulang jika ada perubahan field.
  • 9.1.8 Finalization: Wizard partial lock, close batch, post final ke backend.
  • 9.1.9 Correction Flow: Rollback state attendance ke approve, batch ke correction.
  • 9.1.10 Reporting & Attachment: Slip PDF, 1721 CSV, E-banking.

9.2 Attendance Extension Requirements

Modul: if_hr_payroll_attendance.
Tujuan: Menjadikan payroll sensitif terhadap actual working days, overtime, unpaid leave. Menghitung deduction dari absence, lock_payroll flag untuk menolak run jika ada data belum approve.

9.3 Payroll Journal Requirements

Modul: if_hr_payroll_journal.
Tujuan: Menghasilkan account move / jurnal akuntansi dari batch saat ditutup, memetakan BPJS/PPh21 ke debit/kredit.

9.4 Loan Requirements

Modul: if_loan.
Tujuan: Meng-inject cicilan bulan berjalan ke deduction draft payslip. Menandai installment paid, rollback ke unpaid jika payslip dihapus.

9.5 Mobile Payroll Requirements

Modul: if_mobile_payroll.
Menghasilkan attachment PDF payslip yang dilindungi token.

9.6 Master Company

Tambahan field penting: THR eligibility days, THR round days, THR components, Tax signer name / NPWP, NPP / ID TKU / bupot identity, Cutoff payroll/attendance, Flag overwrite BPJS, Payroll deduction absence components. Behavior: Perubahan field memicu sync company. Create company membuat THR default.

9.7 Government Regulation

Model utama: hr.government.regulation, dll. Dibuat per company, disinkronkan ke backend. Mendukung history effective-dated change.

9.8 Employee

Field tambahan: salary payment journal, payment method, pesangon, BPJS tick date, tax report year. Create/write memicu sync jika field wajib terpenuhi.

9.9 Contract

Field penting: active_backpay, effective_date_backpay, backpay_component_ids, contract_number. Contract adalah unit sinkronisasi utama. Payload berisi basic salary, UMR, cutoff type, dll.

9.10 Payslip Period & Accumulated Payroll

Period dibuat otomatis per tahun. Accumulated Payroll berfungsi untuk inisialisasi / beginning balance.

9.11 Payslip & Payslip Batch

Payslip menyimpan hasil akhir: THP, PPh21, BPJS, actual_working_days. Batch memuat payslip period, state draft/close, lock_payroll flag.

10. Business Rules

  • 10.1 Field Wajib Employee: NPWP, marital status, employment status, payroll method.
  • 10.2 Contract Validity: Hanya status 'open' yang diproses.
  • 10.3 Batch Exclusivity & Finalization Rule: Tak boleh overlap batch reguler. Harus sinkron period backend sebelum action_post_finalized_data().
  • 10.4 Correction & Attendance Rule: Attendance period locked tidak boleh diedit.
  • 10.5 BPJS, THR, Pesangon: Overwrite jika dizinkan company, batch terpisah.

11. UX, Security, Scheduler & Integrasi Eksternal

11.1 UX / Antarmuka

Menyederhanakan menu native, rombak payslip form, menggunakan Wizard-centric Flow (untuk finalize, report, bupot, bpjs, dll).

11.2 Security Requirements

Membedakan grup hak akses. API config hanya untuk base.group_no_one, dll.

11.3 Scheduler / Automation

Generate period per tahun, cron history change, fungsi install awal data.

11.4 Integrasi Eksternal

Payroll API Backend (endpoints /configs, /employees, /contracts, /calculate, /finalize). Koneksi iLovePDF untuk 1721 Excel to PDF. Export banking (BCA, BRI, Mandiri, CIMB).

12. Risiko Produk, Data Kritis & Acceptance Criteria

Risiko Teknis: Ketergantungan API host, SQL manual.
Risiko Operasional: Recalculation terpicu diam-diam jika master diubah, attendance lock macet jika SPV telat approve, error cicilan loan jika draft payslip dihapus sembarangan.
Risiko Compliance: Data report NPWP bocor jika sinkron tidak solid.

12.1 Kebutuhan Data Kritis

API host, config payroll, regulasi govt per company, approval attendance, loan master komplit sebelum run.

12.2 Acceptance Criteria

  • Setup berhasil tanpa error.
  • Regular dan Off-cycle batch dapat membuat draft.
  • Recalculation berhasil menyesuaikan angka jika komponen irregular diubah.
  • Attendance & Loan berefek langsung ke THP draft.
  • Finalize menutup batch dan menciptakan Journal Account Move.

12.3 Open Questions / Gap Implementasi

Default API bernilai "/", butuh initial manual setup. Beberapa placeholder fungsi masih ada. Integration heavy testing.

12.4 Rekomendasi Implementasi Operasional

Urutan SOP: sync company -> govt -> employee -> contract. Wajibkan cut-off approval absensi secara disiplin sebelum batch draft dibuat. Testing UAT di siklus Desember.

14. Alur Payroll Detail & Skenario UAT

Bagian ini merinci langkah prosedural secara detail dari hulu ke hilir untuk siklus gaji bulanan (Regular Payroll).

14.1 Sinkronisasi Master Data

  • Setup API: SysAdmin mengonfigurasi host Odoo pada hr.payroll.api.config. Sistem melakukan test connection.
  • Sync Company: Data risk JKK, Prorate Type, dan batas Cutoff dikirim. Terjadi otomatis saat ada perubahan write() pada Model Company terkait.
  • Sync Employee & Contract: Profil karyawan (NPWP, Status Nikah) dan isi Contract (Basic Salary, Allowance/Deduction tetap) dipastikan harus berstatus Open/Active.

14.2 Persiapan Kehadiran (Pre-Payroll)

  • Supervisor memastikan modul Attendance telah meng-_approve_ semua data cuti, jam lembur, dan absensi.
  • Modul ekstensi if_hr_payroll_attendance akan membaca cut-off kehadiran dan memastikan **tidak ada** lock constraint yang dilanggar.

14.3 Pembuatan Batch & Kalkulasi API

  • HR membuka Payslip Batches dan membuat record baru berdasarkan periode.
  • Odoo memaketkan data (Period ID, Array Contract ID) dan mengirim request POST /calculate ke Backend API Eksternal.
  • API mengembalikan JSON berisi penjabaran BPJS, PPh21, gross, dan THP. Menjadi draft hr.payslip.

14.4 Review, Adjustment & Recalculation

  • HR menambahkan Irregular Component di form Odoo. Modul Ekstensi Loan (if_loan) otomatis membaca hutang aktif cicilan bulan berjalan dan "meng-inject".
  • HR menekan tombol Recalculate. Odoo menembak ulang API. API merespons balik nilai pajak yang direvisi.

14.5 Finalisasi Batch & Accounting

  • HR memilih Finalize/Close Batches via Wizard. Odoo mengirim action_post_finalized_data() ke API.
  • Modul Ekstensi Journal (if_hr_payroll_journal) menangkap state Close ini dan merilis account.move.

14.6 Mekanisme Kunci Attendance (Attendance Lock)

Sebelum payroll batch dapat diproses, sistem menerapkan mekanisme kunci attendance untuk memastikan semua data kehadiran sudah final dan tidak dapat diubah. Mekanisme ini bekerja pada level hr.payslip.run dan mencegah payroll dijalankan jika masih ada data attendance, overtime, atau leave yang berstatus draft, check, atau confirm.

Jika data tersebut ditemukan, sistem membuka form wizard hr.approve.all (via modul if_hr_payroll_attendance) yang memungkinkan supervisor atau admin untuk menyetujui semua data sekaligus sebelum payroll dilanjutkan.

14.7 Validasi Sebelum Finalisasi Batch

Sebelum payslip batch dapat difinalisasi, sistem menjalankan _check_batch_all_data_need_approval() pada hr.payslip.run. Fungsi ini mengecek 3 jenis data per periode attendance:

  • Attendance: Memastikan semua attendance dengan check_in + 7 hours berada dalam rentang periode attendance TIDAK memiliki state draft atau check. Jika ada, fungsi mengembalikan True.
  • Overtime: Memastikan semua overtime dengan check_in + 7 hours dalam rentang TIDAK memiliki state draft atau check.
  • Leave: Memastikan semua leave dengan request_date_from dalam rentang TIDAK memiliki state draft, confirm, atau validate1.

14.8 User Interface — Approve All

Jika need_approve_data=True, sistem membuka form wizard hr.approve.all (via if_hr_payroll_attendance) yang memungkinkan user menyetujui semua data sekaligus. Context bypass_approve dapat diset untuk melewati validasi ini (digunakan pada off-cycle payroll).

14.9 Cutoff Date Logic

Attendance dan payroll menggunakan cutoff date yang ditentukan oleh company.get_company_cutoff_date(year, month, attendance=True/False). Untuk attendance=True, cutoff typically adalah tanggal akhir bulan. Date_from dihitung sebagai cutoff_date + relativedelta(months=-1, days=+1) (hari pertama bulan sebelumnya).

Contoh: cutoff=2026-02-28, from_date=2026-01-29 (untuk periode Januari).

14.10 Attendance State Machine

  • draft → attendance baru dibuat oleh machine/superuser
  • check → attendance di-review oleh SPV
  • approve → attendance siap untuk payroll (finalized)
  • payroll → attendance sudah terkait payslip

Attendances dengan state approve atau payroll tidak dapat diedit ulang kecuali melalui mekanisme payroll correction.

14.11 State Rollback on Payslip Unlink

Ketika payslip di-unlink (dihapus), sistem otomatis me-rollback attendance state dari payroll/approve ke approve/approve dengan menset is_waiting_to_pay=True dan payslip_run_id=False. Ini memastikan attendance dapat diapprove ulang tanpa perlu diedit manual.

Modul journal mendengarkan event ini dan secara otomatis memulai proses pembuatan jurnal akuntansi untuk seluruh payslip yang sudah difinalisasi.
4.12 Journal Module Generate Account Move (Jurnal BPJS, PPh21, THP) Modul if_hr_payroll_journal membuat Account Move (jurnal akuntansi) untuk setiap komponen payroll. Jurnal yang dibuat biasanya mencakup: (a) Jurnal Gaji (THP) — debit: biaya gaji, kredit: hutang gaji/bank, (b) Jurnal BPJS Kesehatan & Ketenagakerjaan — debit: biaya BPJS perusahaan, kredit: hutang BPJS, (c) Jurnal PPh21 — debit: biaya pajak karyawan, kredit: hutang PPh21, (d) Jurnal Iuran BPJS Karyawan — debit: hutang gaji, kredit: hutang BPJS. Setiap Account Move di-post untuk mengubah posisi account di general ledger.
4.13 Odoo (Master & Batch) Generate PDF Payslip + Token Sistem menghasilkan dokumen PDF payslip untuk setiap karyawan yang sudah difinalisasi. PDF ini berisi slip gaji lengkap dengan seluruh komponen: gaji pokok, tunjangan, potongan, PPh21, BPJS, dan take home pay. Sistem juga men-generate token unik untuk setiap payslip. Token ini digunakan oleh modul if_mobile_payroll untuk mekanisme unduh PDF payslip melalui aplikasi mobile — karyawan bisa mengakses payslip mereka secara secure menggunakan token tanpa perlu login ke Odoo.
4.14 Odoo (Master & Batch) recalculate_concurrent_batch() Ini adalah langkah terakhir yang sangat penting. Setelah satu batch difinalisasi, sistem memanggil recalculate_concurrent_batch() untuk mengecek apakah ada batch payroll lain yang masih berstatus Draft, Submit, atau Correction yang perlu di-recalculate. Jika ada, sistem akan mengulangi proses create_payslip_data() untuk batch-batch tersebut agar data payroll tetap konsisten. Fungsi ini juga mereset flag active_backpay=False pada kontrak yang sudah diproses di batch lain. Ini memastikan bahwa payroll tidak tumpang tindih dan setiap batch terupdate dengan data terakhir.

17. Alur Teknis Payroll Correction, Loan Injection & Mobile Payslip

17.1 Kondisi Pemicu Correction

Correction dibutuhkan ketika:

  • Employee belum termasuk di batch regular (missed input)
  • Ada kesalahan input komponen gaji (allowance/deduction)
  • Ada perubahan data employee yang mempengaruhi payroll SETELAH batch difinalisasi
  • Payroll salah bulan (misalnya di-assign ke periode yang salah)

17.2 Langkah Teknis Correction

  1. Buka Batch: Batch kembali ke state draft (jika state memungkinkan) menggunakan wizard atau button khusus.
  2. Rollback Attendance: Semua attendance terkait dikembalikan ke state approve dengan is_waiting_to_pay=True. Lihat HrPayslip.unlink().
  3. Rollback Overtime: Sama dengan attendance, overtime dikembalikan ke state approve.
  4. Hapus Payslip: Payslip employee yang dikoreksi di-unlink. Proses ini trigger rollback attendance dan overtime secara otomatis.
  5. Input Koreksi: Data payroll yang benar di-input via wizard atau import XLSX.
  6. Reproses Batch: Batch diproses ulang, attendance diapprove ulang, dan payslip dihitung ulang.
  7. Repost Journal: Account move entries yang lama di-reverse (jika applicable) dan entries baru di-post.

17.3 Mekanisme Context Bypass

Correction menggunakan context = {'bypass_lock_payroll': True, 'bypass_approve': True} untuk melewati:

  • bypass_lock_payroll: melewati validasi payslip lock (state=done / lock_payroll=True)
  • bypass_approve: melewati validasi data approval attendance/leave/overtime

17.6 Injeksi Otomatis ke Payslip

Setiap kali payroll di-compute (sync_salary_computation()), sistem memproses pinjam aktif employee:

  1. Cari semua hr.loan dengan employee_id=payslip.employee_id, state='approve'
  2. Untuk setiap loan, cek apakah bulan berjalan sudah masuk dalam periode loan (start_date ≤ period_end)
  3. Jika ya, catat installment_amount sebagai hr.payslip.deduction dengan name=loan_type dan amount=installment_amount
  4. Kurangi remaining_amount di record loan
  5. Jika remaining_amount ≤ 0, auto-close loan (state='done')

17.7 Konteks Pengabaian Lock

Pembayaran angsuran loan menggunakan with_context(bypass_lock_payroll=True) agar tidak terblokir oleh payroll lock yang mungkin aktif. Ini memungkinkan loan deduction tetap ter-input meskipun payroll sudah di-lock.

17.8 Alur Mobile Payslip PDF (if_mobile_payroll)

17.9 Mekanisme Generate

Payslip PDF digenerate on-demand saat employee atau admin mengakses payslip dari mobile app. Sistem menggunakan report engine Odoo dengan template QWeb if_mobile_payroll.report_payslip. PDF dilindungi token: URL download mengandung token unik yang expire dalam waktu tertentu.

17.10 Token-Based Security

Setiap employee memiliki token field di record mereka. Token ini digunakan sebagai parameter untuk generate PDF payslip. Token expires setelah periode tertentu (konfigurasi di hr.payroll.api.config). Jika token tidak valid atau expired, akses ditolak dengan error response.

17.11 Data Payslip di Mobile

Mobile app consuming data via REST API endpoint /api/payslip/mobile/[id]. Response JSON mengandung: employee info, period, computed_wage, allowance breakdown (regular/irregular), deduction breakdown (BPJS/health, tax, loan, dll.), take_home_pay, dan URL signed untuk PDF attachment.

19.2 Record Rules

  • hr.payslip: Employee hanya bisa read/edit payslip miliknya sendiri (employee_id = uid.employee_id)
  • hr.attendance: SPV bisa approve attendance milik department-nya saja
  • hr.loan: Employee bisa read loans miliknya, HR/Admin bisa CRUD semua
  • hr.payroll.api.config: Admin only — tidak bisa di-read oleh user biasa (credential/api key)
  • hr.api.log: Admin + Payroll Admin bisa read

19.3 Field-Level Security

  • hr.employee.bpjs_health_number: HR Admin + Employee (own record) can read
  • hr.employee.npwp: HR Admin + Employee (own record) can read
  • hr.payroll.api.config: Semua API credential fields — Admin only write, no read untuk others
  • hr.contract.wage: HR Manager + Payroll Admin can write
  • hr.payslip.computed_wage: Auto-computed, no manual write untuk employee

19.4 Context Bypass Usage Policy

Context bypass_lock_payroll dan bypass_approve adalah elevated privileges yang HARUS:

  • Hanya digunakan oleh Payroll Admin atau System Scheduler
  • Selalu accompanied dengan audit log di hr.api.log
  • Tidak boleh digunakan untuk routine employee self-service flows
  • Di-review quarterly oleh IT Security

Model: hr.government.regulation | Method: sync_*() family

Model: hr.employee | Method: _apply_scheduled_marital_tax_status_update()
Frequency: Annual (January 1, 00:05)
Deskripsi: Mengeksekusi semua scheduled marital status update yang effective_date=1 Januari tahun berjalan. Membuat history.change records. Langsung mengapply PTKP baru untuk payroll tahun berjalan.

Model: hr.payslip (commented out) | Method: reset calculated_amount on hr.contract.irregular.allowance

Deskripsi: Me-reset calculated_amount pada irregular allowances (khusus yang basic_calculation!='contract') setelah payslip difinalize. Mempersiapkan kalkulasi bulan berikutnya.

Untuk deployment, pastikan semua modul fp_hr_payroll, if_hr_payroll_attendance, if_hr_payroll_journal, if_loan, dan if_mobile_payroll terinstal berurutan sesuai dependency. Konfigurasi API endpoint dan credential dilakukan via hr.payroll.api.config per company. Scheduler harus diaktifkan di menu Settings > Technical > Automation > Scheduled Actions.
