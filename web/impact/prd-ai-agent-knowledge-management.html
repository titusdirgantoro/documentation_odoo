AI Agent Knowledge Management
Berbasis Telegram Bot untuk Odoo 17

Sistem asisten virtual berbasis RAG yang memungkinkan tim Business Analyst mengakses pengetahuan teknis Odoo 17 secara cepat, akurat, dan aman dari kebocoran data.

1

Rangkuman Eksekutif

Proyek ini membangun asisten virtual berbasis Telegram yang ditenagai AI dengan teknologi Retrieval-Augmented Generation (RAG). Sistem ini dilatih menggunakan dokumentasi Odoo 17, struktur kode base, dan custom module internal — semuanya berjalan 100% di server internal perusahaan demi menjaga keamanan aset intelektual. Tujuan utama: memberdayakan tim Business Analyst (BA) agar dapat mengekstrak informasi teknis, memvalidasi fitur, dan menyusun PRD tanpa harus membaca ribuan baris kode secara manual.

🎯 Masalah yang Dipecahkan

  • BA kesulitan memahami kode Odoo tanpa bantuan developer
  • Bottleneck pengetahuan teknis menghambat pembuatan PRD
  • Tidak ada sistem terpusat untuk query dokumentasi Odoo
  • Risiko kebocoran kode custom ke platform cloud pihak ketiga

💡 Solusi yang Diusulkan

  • Dify.ai self-hosted sebagai AI orchestration platform
  • Vector database lokal untuk semua knowledge base kode
  • Telegram Bot sebagai antarmuka percakapan per-user
  • Hybrid RAG: dense + BM25 search untuk akurasi kode

Kebutuhan Bisnis

IMPORTANT

📋 Functional Requirements

  • FR-01: Sistem mampu menjawab pertanyaan teknis berbasis dokumentasi Odoo 17 (PDF/HTML) dengan akurasi ≥ 85%
  • FR-02: Sistem mampu melakukan retrieval atas kode Python/XML dari custom module internal dengan recall ≥ 80%
  • FR-03: Setiap user Telegram memiliki session memory percakapan independen (minimal 20 pesan terakhir)
  • FR-04: Bot merespons dalam bahasa Indonesia dan Inggris sesuai input user
  • FR-05: Sistem mendukung upload dokumen baru ke knowledge base tanpa downtime total
  • FR-06: Tersedia fitur feedback (👍/👎) untuk setiap jawaban bot guna evaluasi kualitas

🔒 Non-Functional Requirements

  • NFR-01 (Keamanan): Source code custom module TIDAK BOLEH meninggalkan infrastruktur internal (zero cloud upload untuk raw code)
  • NFR-02 (Ketersediaan): Uptime minimal 95% pada jam kerja (08:00–18:00 WIB, Senin–Jumat)
  • NFR-03 (Performa): Response time end-to-end ≤ 10 detik untuk 95% query
  • NFR-04 (Skalabilitas): Mampu menangani minimal 10 concurrent users tanpa degradasi signifikan
  • NFR-05 (Audit): Semua query dan jawaban tercatat dalam log untuk audit 90 hari ke belakang
  • NFR-06 (Legal): Kontrak DPA (Data Processing Agreement) ditandatangani dengan LLM provider (Anthropic/OpenAI) sebelum go-live

🔴
Gap Kritis #1: Chunking Strategy untuk Kode Python

Chunking naif (512 karakter) akan menghancurkan konteks semantik fungsi Odoo. Wajib menggunakan AST-aware chunking berbasis Python ast module atau tree-sitter: pisahkan per-fungsi/per-class. Terapkan Hybrid Search: Dense Vector (semantic) + BM25 (untuk nomenclature seperti mrp_production, stock_picking). Indexing diorganisir per-module untuk retrieval terstruktur.

🔧 Komponen Infrastruktur

  • OS: Ubuntu 24.04 LTS
  • Container: Docker Compose (isolated)
  • Dify.ai: nginx + api + worker
  • Vector DB: Qdrant (lebih ringan dari Weaviate)
  • Cache: Redis (session + query cache)
  • Database: PostgreSQL 15

📥 Tiered Ingestion Strategy

  • Tier 1 – Stable Base: Odoo 17 base code (~50K+ files), satu kali ingestion, stabil
  • Tier 2 – Living: Custom modules, re-ingestion saat ada commit baru ke repo
  • Tier 3 – Docs: PDF/HTML dokumentasi Odoo, update per rilis versi
  • Proses re-ingestion dijadwalkan via cron job otomatis

Kebutuhan Operasional

ADDED
🔴
Gap Kritis #2: Ops Runbook — Wajib Ada Sebelum Go-Live

Dify Community Edition tidak memiliki auto-update, high availability, atau built-in monitoring. Ops runbook harus mencakup: siapa PIC maintenance, prosedur backup vector DB, prosedur re-ingestion saat ada update kode, dan prosedur rollback jika ada kegagalan.

👥 Struktur Tim Operasional

  • PIC DevOps (1 orang): Bertanggung jawab atas health server, backup, update Dify, dan re-ingestion terjadwal
  • PIC AI/QA (1 orang): Memantau akurasi jawaban bot, mengelola feedback loop, koordinasi evaluasi bulanan
  • Escalation: Bila akurasi turun di bawah 75%, eskalasi ke developer senior untuk audit knowledge base

🔄 Backup & Recovery Strategy

  • Vector DB Backup: Snapshot Qdrant harian, disimpan di storage terpisah (retention 30 hari)
  • PostgreSQL Backup: pg_dump otomatis setiap malam (retention 14 hari)
  • Dify Config Backup: Export workflow/agent config ke Git repository internal
  • RTO Target: ≤ 4 jam (recovery dari backup)
  • RPO Target: ≤ 24 jam (maksimum data yang dapat hilang)

📅 Jadwal Maintenance Rutin

  • Harian (Otomatis): Backup data, health check endpoint, monitoring resource usage
  • Mingguan (Manual, <1 jam): Review log anomali, cek token usage vs budget, update jika ada patch kritis
  • Per Sprint/Deploy Custom Module: Trigger re-ingestion Tier 2 (custom modules) — setelah QA sign-off
  • Bulanan (2-4 jam): Evaluasi KPI, review feedback loop, pertimbangkan tuning retrieval strategy
  • Per Rilis Odoo: Re-ingestion Tier 1 (base code) jika ada update versi signifikan

Legal & Compliance

CRITICAL
🔴
Gap Kritis #3: Klarifikasi Data ke LLM Provider

Meskipun raw source code tidak pernah diunggah, potongan kode (chunks) relevan dikirim ke API LLM sebagai konteks untuk setiap query. Ini perlu diklarifikasi dengan legal team: apakah Anthropic/OpenAI dianggap "acceptable third party"? Kedua provider memiliki DPA enterprise — pastikan tidak ada data training opt-in yang aktif.

✅ Checklist Legal Sebelum Go-Live

  • Tanda tangan DPA dengan Anthropic (Claude API) atau OpenAI
  • Konfirmasi "Zero Data Retention" / opsi opt-out dari training data
  • Review dengan legal team: apakah chunks kode masuk kategori "data rahasia" per perjanjian kerja karyawan
  • Dokumentasi data flow (dari server internal → API → response) untuk audit trail
  • Pertimbangkan alternatif fully on-premise: Ollama + Llama 3/CodeLlama jika legal tidak menyetujui API eksternal

🛡 Alternatif Fully On-Premise (Opsional)

  • Ollama + CodeLlama 34B: LLM berjalan 100% lokal, nol data keluar jaringan
  • Trade-off: Kualitas jawaban lebih rendah dari Claude 3.5 Sonnet, butuh GPU/CPU beefy (min. 32GB RAM untuk 34B)
  • Biaya hardware: Signifikan lebih tinggi (server dengan GPU atau bare-metal)
  • Rekomendasi: Mulai dengan Claude API + DPA. Evaluasi on-premise di akhir fase 1 jika ada keberatan legal.
