Membangun Transformation OS — sistem yang membantu manajer membaca data, mengambil tindakan, memantau hasil, dan menyimpan pelajaran agar perbaikan bisnis bisa diulang.

⚙️ Perbedaan Utama dari Doc A

Section 2
Schema Mapping Layer — Menangani Format File Acak
Setiap user punya template Excel yang berbeda. Schema Mapping Layer memungkinkan sistem belajar format tersebut sekali saja, lalu semua upload berikutnya berjalan deterministik.
💡 Catatan penting: Ini adalah satu-satunya tempat AI digunakan dalam pipeline data — dan hanya terjadi sekali per template baru. Setelah mapping tersimpan, tidak ada AI lagi dalam proses normalisasi data.

1

Trigger: Upload File

User upload .xlsx, .csv, atau data masuk dari Odoo sync. Sistem baca template_hash dari header file.

2

Cek column_mappings

Jika template_hash sudah ada di DB → langsung ke parsing. Jika belum → trigger AI Schema Discovery.

3

AI Schema Discovery (hanya jika template baru)

Claude membaca header + 5 baris pertama. Mengusulkan mapping kolom ke canonical schema. Tidak membaca seluruh file.

4

User Validasi & Konfirmasi

UI menampilkan mapping untuk dikonfirmasi user. User boleh koreksi per kolom. Setelah dikonfirmasi, mapping disimpan permanen.

5

Deterministic Parser Berjalan

Data dibaca menggunakan saved mapping. Standarisasi tanggal, satuan, dan validasi tipe data dilakukan oleh library (pandas/xlsx) — bukan AI.

6

INSERT ke Canonical Facts

Data bersih dimasukkan ke tabel facts_sales_daily, facts_hr_daily, facts_inventory_daily sesuai departemen.

Section 4
Metric Engine — Perhitungan KPI Deterministik
Jantung dari Doc B. Semua KPI dihitung oleh sistem dengan formula yang terdefinisi jelas. Bukan oleh AI. Hasilnya konsisten, bisa diaudit, dan bisa dipercaya.
Alur: Canonical Facts → KPI Calculation → Metric Snapshot

Section 7
Learning Loop — What Worked & Playbook Builder
Sistem tidak hanya menyarankan task — ia juga memantau apakah task itu benar-benar memperbaiki KPI target. Hasilnya direkam sebagai pembelajaran yang bisa diulang.
Learning Loop — Dari Task Suggestion ke Playbook
sequenceDiagram participant AE as 📈 Analytics Engine participant AI as 🤖 Claude AI participant MGR as 👤 Manager participant SYS as ⚙️ Sistem participant DB as 🗄️ Database AE->>AI: Kirim metric snapshot: absenteeism 10%, Priority AI-->>MGR: Suggest task: "Coaching 1-on-1 dengan 2 staf absen" MGR->>SYS: Setujui task, assign ke HR, deadline 2 minggu SYS->>DB: INSERT tasks (title, target_kpi=absenteeism, assignee, deadline) Note over SYS,DB: Setelah deadline lewat — sistem verifikasi otomatis SYS->>DB: SELECT metric_snapshots WHERE metric=absenteeism AND period AFTER task_created SYS->>SYS: Bandingkan: absenteeism sebelum = 10%, sesudah = 5.5% SYS->>DB: INSERT task_impact_results (before: 10%, after: 5.5%, result: improved) alt KPI membaik SYS->>DB: INSERT what_worked (task, context, kpi_change, dept) SYS->>AI: Trigger: susun playbook jika pola ini berhasil 3x AI-->>DB: INSERT playbooks (situation, action, expected_outcome) else KPI tidak membaik SYS->>DB: INSERT what_didnt_work (task, context, probable_reason) Note over SYS: Data ini membantu AI tidak menyarankan task serupa end

✅ what_worked

Rekaman task yang berhasil memperbaiki KPI target. Berisi konteks departemen, kondisi KPI sebelum dan sesudah, dan rentang waktu pemulihan.

❌ what_didnt_work

Rekaman task yang tidak berdampak atau memperburuk KPI. Digunakan AI untuk menghindari saran serupa di masa depan pada kondisi yang sama.

📋 Playbooks

Dihasilkan AI saat pola yang sama berhasil 3 kali atau lebih. Format: Situasi → Aksi → Hasil yang Diharapkan. Bisa dibagikan antar departemen.

Aturan Sistem — Non-Negotiable Guardrails
Aturan ini tidak boleh dilanggar dalam implementasi apapun. Mereka adalah fondasi kepercayaan sistem.

Isolasi Multi-Tenant Mutlak WAJIB — TIDAK BOLEH DILANGGAR

Setiap query ke database HARUS menyertakan org_id. Tidak ada mode superadmin yang bisa bypass isolasi ini. Row-Level Security PostgreSQL aktif sebagai pengaman tambahan.

org_id filterRLS PostgreSQLNo bypass

Semua KPI Harus Ada Formula Tertulis WAJIB

Tidak ada KPI yang muncul dari output AI langsung. Setiap metrik harus terdefinisi di tabel metric_definitions dengan formula yang jelas, bisa ditunjukkan ke user kapan saja.

metric_definitionsAuditable formula
🤖

AI Tidak Boleh Baca File Mentah untuk Perhitungan WAJIB — TIDAK BOLEH DILANGGAR

AI (Claude) hanya diizinkan membaca header + 5 baris sample untuk schema discovery. Untuk task lain, AI hanya menerima JSON terstruktur dari Analytics Engine — bukan file raw.

5 baris max untuk mappingJSON only untuk analitik

AI Suggestion Harus Memiliki KPI Target Spesifik WAJIB

Setiap task suggestion dari AI wajib menyebut target_kpi yang spesifik (contoh: absenteeism_rate, bukan hanya "perbaiki kehadiran"). Ini memungkinkan sistem mengukur dampak task secara deterministik.

target_kpi requiredMeasurable outcome

Playbook Hanya Diterbitkan Setelah 3 Kali Berhasil ANJURAN KUAT

Sistem tidak boleh mempromosikan sebuah pola sebagai "playbook" hanya berdasarkan satu keberhasilan. Minimal 3 instance berhasil di kondisi yang comparable sebelum diterbitkan ke playbook library.
