OWL Frontend Framework: Evolusi

Perbandingan mendalam OWL 1.x (Odoo 17) vs OWL 2 + Vue.js (Odoo 19)

⚙️ Odoo 17 — OWL 1.x

  • Component-based, inspired by React/Vue
  • Manual state management: this.render() harus dipanggil
  • Lifecycle: setup(), onWillStart(), onMounted(), onWillUpdateProps(), onPatched(), onWillUnmount()
  • Reactive arrays harus di-replace seluruhnya untuk trigger re-render
  • Hooks dasar: useState, useRef, useEnv
  • QWeb template dengan t-esc (escaping)
  • Asset bundling via Webpack (multiple bundles)
  • JS load: semua assets di-load sekaligus saat startup
  • No built-in lazy loading

🚀 Odoo 19 — OWL 2 + Vue.js

  • Fully reactive: DOM otomatis update saat state berubah
  • NO manual this.render() — reactive arrays auto-trigger
  • New hooks: useAssets (lazy load), useAutofocus, useBus (auto cleanup), usePosition, useSpellCheck
  • async/await, ES6 destructuring, spread operators native
  • Vue.js integration — component reusability tinggi
  • QWeb template dengan t-out (new standard)
  • Dynamic asset loading — load hanya saat dibutuhkan
  • Frontend bundles lebih kecil → halaman lebih cepat
  • OWL Migration di website/e-commerce → rendering speed ↑
💡
Code Migration Example: Di Odoo 17, kamu harus tulis this.state.items = [...this.state.items, newItem]; this.render(); — di Odoo 19 cukup this.state.items.push(newItem); dan DOM update otomatis.

⚙️ Odoo 17 — Component Code Pattern

// Odoo 17 OWL 1.x Component import { Component, useState } from "@odoo/owl"; export class MyWidget extends Component { setup() { this.state = useState({ items: [], loading: false }); } async loadData() { this.state.loading = true; const data = await fetchData(); // ⚠️ Must replace entire array this.state.items = [...data]; this.state.loading = false; // ⚠️ Manual render call required this.render(); } }

🚀 Odoo 19 — Component Code Pattern

// Odoo 19 OWL 2 Component import { Component, useState, useAssets } from "@odoo/owl"; export class MyWidget extends Component { setup() { this.state = useState({ items: [], loading: false }); // ✅ Lazy-load assets useAssets({ jsLibs: ["/my/chart.js"] }); } async loadData() { this.state.loading = true; const data = await fetchData(); // ✅ Direct push — auto re-render! this.state.items.push(...data); this.state.loading = false; // ✅ No render() call needed } }

ORM & Performance

Perubahan ORM, database performance, dan optimisasi query

⚙️ ORM Odoo 17

ORM matang dan stabil. Standard CRUD dengan psycopg2.

  • search() dengan inselect (subquery IN SELECT)
  • Standard B-tree indexing
  • Computed fields: re-compute on-demand
  • Binary fields: load lazily tapi cache terbatas
  • Prefetch: partial batch prefetching
  • stock.valuation.layer untuk inventory value
  • PostgreSQL 12+ support

🚀 ORM Odoo 19 (+30-40% faster)

New query planner + batch prefetch + binary caching.

  • SQL IN clause menggantikan inselect → query lebih efisien
  • New query planner: smart join optimization
  • Batch prefetching — kurangi N+1 queries
  • Improved binary field caching
  • Enhanced computed field caching
  • stock.valuation.layer → dihapus, pindah ke stock.move
  • PostgreSQL 14–16: parallel query ↑↑

  • Basic email routing suggestions
  • Simple lead scoring (rule-based)
  • Automated bank reconciliation

API Architecture: JSON-RPC → REST

Dari JSON-RPC monolitik ke Enterprise REST API dengan OAuth2 dan WebSocket

⚙️ Odoo 17 — JSON-RPC

  • Primary: JSON-RPC 2.0
  • Satu endpoint: /web/dataset/call_kw
  • Auth: session-based (cookies)
  • API Key untuk external calls
  • XML-RPC tersedia (legacy)
  • Tidak ada native OAuth2
  • Real-time: long-polling
  • Response: JSON-wrapped format

🚀 Odoo 19 — Enterprise REST

  • Native RESTful per resource
  • Endpoint bersih: /api/v1/model.name
  • OAuth2 + Token authentication
  • HTTP verbs: GET/POST/PUT/DELETE
  • WebSocket real-time events
  • API versioning (/v1/, /v2/)
  • OpenAPI / Swagger docs
  • Rate limiting + Webhook native

⚙️ API Call — Odoo 17

# JSON-RPC call Odoo 17 payload = { "jsonrpc": "2.0", "method": "call", "params": { "model": "res.partner", "method": "search_read", "args": [[("is_company","=",True)]], "kwargs": {"fields": ["name","email"]} } } r = requests.post( f"{url}/web/dataset/call_kw", json=payload, cookies=session)

🚀 API Call — Odoo 19

# REST API Odoo 19 — lebih bersih! headers = { "Authorization": f"Bearer {token}", "Content-Type": "application/json" } # Standard REST endpoint r = requests.get( f"{url}/api/v1/res.partner", params={"filter": '[("is_company","=",true)]', "fields": "name,email"}, headers=headers) partners = r.json()

Panduan Migrasi: Odoo 17 → Odoo 19

Apa yang harus diperhatikan saat upgrade versi

✅ Auto-migrate

Modul standar, konfigurasi, data master — migrasi otomatis oleh Odoo upgrade tool.

⚠️ Perlu Review

Custom XML views, JS overrides, t-esc templates, API endpoint changes.

🔴 Breaking Changes

stock.valuation.layer dihapus, xmlrpc_port renamed, OWL 1.x manual render, JSON-RPC integrations.

✅ Python OK

Code Python 3.10+ sudah compatible langsung ke Odoo 19. Optimal di 3.12.

⚠️ PostgreSQL Upgrade

Perlu upgrade dari pg12 → pg14/16. Data migration aman dengan pg_upgrade.

🔴 OWL JS Components

Custom OWL 1.x components perlu diport: this.render() → reactive state pattern OWL 2.

✅ Business Logic

Model methods Python (@api.model, @api.depends) mayoritas tetap compatible.

⚠️ QWeb Templates

t-esc masih berjalan tapi deprecated. Sebaiknya migrate ke t-out secara bertahap.

🔴 External API Clients

Sistem eksternal berbasis JSON-RPC harus diupdate ke REST API + OAuth2 flow baru.

📋 Roadmap Migrasi (Estimasi Timeline)

Phase 1 — Assessment (1–2 minggu)

  • Audit semua custom JS — cari this.render() calls
  • List semua external integrations berbasis JSON-RPC
  • Cek apakah ada stock.valuation.layer di custom code
  • Periksa xmlrpc_port dalam semua config files

Phase 2 — Dev Environment (2–4 minggu)

  • Setup Odoo 19 dev environment (Python 3.10+, PostgreSQL 14+)
  • Run odoo-upgrade tool untuk auto-migrate database
  • Identifikasi dan fix breaking errors dari custom modules

Phase 3 — Code Migration (4–8 minggu)

  • Port OWL 1.x → OWL 2 reactive state pattern
  • Ganti t-esc → t-out di custom QWeb templates
  • Update external integrations ke REST API + OAuth2
  • Migrate inventory valuation ke stock.move

Phase 4 — Testing & Go Live (2–4 minggu)

  • Full regression testing semua workflows
  • Performance benchmark dengan data production
  • User training: dark mode, sidebar nav, AI features baru
  • Cutover: backup → downtime → switch → monitor

Kapan Harus Upgrade?

Framework keputusan: bertahan di v17 LTS vs upgrade ke v19

🏠 Tetap di Odoo 17 LTS jika...

  • Sistem stabil, tim puas dengan fitur yang ada
  • Banyak kustomisasi kompleks, butuh effort besar untuk port
  • Budget terbatas untuk upgrade tahun ini
  • Tidak butuh fitur AI yang advanced
  • Banyak integrasi eksternal berbasis JSON-RPC
  • Team belum familiar dengan OWL 2 / Vue.js
  • Mau nunggu Odoo 19 lebih mature (6 bulan post-release)

🚀 Upgrade ke Odoo 19 jika...

  • Ingin leverage AI automation untuk efisiensi operasional
  • Perlu performa lebih baik untuk data besar / user banyak
  • Butuh WebSocket real-time (dashboard, notifikasi live)
  • Tim developer ingin modern tooling (OWL 2, REST API)
  • Butuh mobile experience yang jauh lebih baik
  • Greenfield project / implementasi baru
  • Perlu ESG reporting / compliance tools CSRD
📌
Rekomendasi Akhir: Untuk bisnis yang stabil di Odoo 17, tunggu minimal 6 bulan setelah Odoo 19 rilis resmi (September 2025) sebelum migrasi production — untuk memastikan ekosistem module dan community patches sudah matang. Odoo 17 LTS masih fully supported hingga akhir 2026.
