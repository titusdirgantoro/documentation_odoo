Penjelasan Setiap Model

Modul security_seal_serial terdiri dari 5 model custom baru dan 3 model Odoo standar yang di-extend. Berikut penjelasan fungsi masing-masing model, field kunci, dan perannya dalam alur bisnis.

Model Custom Utama

seal.serial.range
📋 Range Serial per Sales Order
Model inti dari seluruh modul. Setiap kali customer memesan produk Security Seal, satu record seal.serial.range dibuat untuk menyimpan range serial number yang dialokasikan khusus untuk SO tersebut. Model ini mencegah tumpang-tindih range antar customer maupun antar order dari customer yang sama, dan melacak sisa range yang belum dikirim (remaining segments) saat partial delivery.
Kenapa diperlukan? Odoo standar menggunakan stock.lot per serial → untuk 1.000 pcs harus buat 1.000 record. Model ini cukup 1 record untuk 1.000 pcs dengan menyimpan angka awal dan akhir saja.
prefix (UA, UB, UX) serial_from (int) serial_to (int) state (draft/confirmed/received/partial/delivered/closed) customer_id (res.partner) sale_order_id remaining_segments (JSON) allocation_ids (O2M)
seal.serial.allocation
🚚 Alokasi Serial per Delivery
Setiap kali sebagian (atau seluruh) range dikirim melalui satu Delivery Order, dibuat 1 record seal.serial.allocation. Record ini menghubungkan antara range induk (seal.serial.range) dengan stock.picking (DO) dan mencatat sub-range yang dikirim pada pengiriman tersebut.
Fungsi utama: menjadi dasar validasi retur — saat customer meretur serial tertentu, sistem cek apakah serial tersebut memang ada di salah satu allocation record customer itu.
serial_range_id (M2O) picking_id (stock.picking) serial_from (int) serial_to (int) qty_allocated (int) active_qty (int) state (draft/done/partial_returned/fully_returned)
seal.returned.serial
↩ Header Retur Serial
Model header untuk event pengembalian barang. Setiap kali customer meretur 1 atau lebih serial number, 1 record seal.returned.serial dibuat sebagai "kepala" retur tersebut. Header ini menyimpan informasi DO asal, jumlah total yang dikembalikan, dan status proses retur (draft → validated → restocked).
allocation_id (M2O) return_picking_id (stock.picking) qty_returned (int) state (draft/validated/restocked) line_ids (O2M → .line) return_date
seal.returned.serial.line
🏷 Detail Serial Diretur (RETIRED)
Model baris dari retur — setiap serial number yang dikembalikan mendapat 1 record di sini. Begitu record ini dibuat dan divalidasi, state-nya langsung menjadi RETIRED secara permanen dan global: serial tersebut tidak dapat digunakan kembali oleh siapapun, termasuk customer berbeda sekalipun.
Inilah "daftar hitam" serial number. Setiap kali ada range baru yang ingin dibuat, sistem akan mengecek tabel ini untuk memastikan tidak ada serial yang sudah RETIRED masuk ke dalam range baru.
returned_serial_id (M2O header) prefix (UA) serial_number (int) serial_display (UA510) state (RETIRED — permanen) retired_date original_customer_id
seal.customer.serial.history
📊 Histori & Saran Next Range
Model ringkasan yang selalu diperbarui setiap kali ada delivery ke customer. Menyimpan serial terakhir yang terkirim (last_serial_number) sehingga pengguna dapat langsung tahu dari nomor berapa order berikutnya harus dimulai — tanpa perlu menelusuri semua SO lama secara manual.
Model ini juga menjadi sumber Smart Suggestion di form Sales Order: sistem otomatis menyarankan nomor awal untuk order baru berdasarkan histori ini.
customer_id (res.partner) product_id prefix last_serial_number (int) total_active_delivered (int) total_returned (int) last_updated

Algoritma Komputasi Sisa Segmen

def compute_remaining_segments(range_rec):
    total = (range_rec.serial_from, range_rec.serial_to)
    allocs = sorted(range_rec.allocation_ids, key=lambda a: a.serial_from)
    segments, current = [], total[0]
    for alloc in allocs:
        if current < alloc.serial_from:
            segments.append((current, alloc.serial_from - 1))
        current = alloc.serial_to + 1
    if current <= total[1]:
        segments.append((current, total[1]))
    return segments
# Setelah DO1 (300-400): → [(1,299),(401,500)]
# Setelah DO2 (1-299)  : → [(401,500)]
# Setelah DO3 (401-500): → []  ✓

seal.serial.range
SR-002
Tetap 1 record untuk keseluruhan range UB001–UB500. Yang berubah adalah field state dan remaining_segments. Setelah DO1: remaining=[(1,299),(401,500)]. Setelah DO2: remaining=[(401,500)]. Setelah DO3: remaining=[], state=delivered. Satu range record ini cukup untuk melacak 3 batch pengiriman berbeda.
seal.serial.allocation
SA-002, 003, 004
3 record allocation, masing-masing merepresentasikan satu pengiriman:
• SA-002: serial_from=300, serial_to=400, qty=101 (DO WH/OUT/00002)
• SA-003: serial_from=1, serial_to=299, qty=299 (DO WH/OUT/00003)
• SA-004: serial_from=401, serial_to=500, qty=100 (DO WH/OUT/00004)
Perhatikan urutan pengiriman tidak harus berurutan (300-400 dikirim sebelum 1-299) — sistem tetap bisa memvalidasi dengan benar karena setiap allocation mencatat sub-range-nya sendiri.
Overlap validation
3x cek
Setiap kali allocation baru dibuat, sistem menjalankan query: "apakah range (from, to) yang baru ini tumpang tindih dengan allocation lain di SR-002?" — ini mencegah pengiriman serial yang sama dua kali ke customer yang berbeda dalam satu batch pengiriman parsial.
💡 Kesimpulan S2: Partial delivery menghasilkan 1 SR + 3 SA records. Jumlah SA = jumlah pengiriman yang dilakukan. SR tetap 1 dan cukup melacak sisa segmen secara efisien menggunakan array JSON.

Inilah poin krusial: pada skenario error, tidak ada satu record pun yang dibuat. Validasi terjadi di layer Python (ORM constraint) sebelum transaksi database di-commit. Jika validasi gagal, Odoo men-raise ValidationError dan seluruh transaksi di-rollback otomatis.
S6A — Overlap check
query SQL
Query yang dijalankan: SELECT id FROM seal_serial_range WHERE customer_id=X AND prefix='UA' AND serial_from<=1500 AND serial_to>=500 AND state NOT IN ('cancelled'). Jika query ini mengembalikan baris apapun, validasi gagal. Hasilnya: SR-001 ditemukan (serial_from=1 ≤ 1500 dan serial_to=1000 ≥ 500) → Error.
S6B — Retired check
cross-customer
Query: SELECT serial_number FROM seal_returned_serial_line WHERE prefix='UA' AND serial_number BETWEEN 500 AND 525 AND state='retired'. Perhatikan: query ini tidak membatasi per customer — RETIRED bersifat global. PT. XYZ pun tidak bisa mendapatkan range yang mengandung UA510 meskipun beda customer dengan PT. ABC.
Error message
user-friendly
Sistem menampilkan pesan yang informatif: menyebutkan record mana yang konflik (SR-001), range mana yang overlap, dan saran perbaikan ("gunakan mulai UA1001"). Ini menghilangkan kebutuhan user untuk "menebak" nomor yang aman.
💡 Kesimpulan S6: Skenario validasi gagal = 0 records dibuat. Validasi terjadi di 2 lapisan: (1) overlap range per-customer, (2) retired serial check lintas customer. Keduanya harus lulus agar record bisa disimpan.

Parsing Serial Number

import re

def parse_serial(serial_str):
    match = re.match(r'^([A-Za-z\-]+)(\d+)$', serial_str.strip())
    if not match:
        raise ValidationError(f"Format serial tidak valid: {serial_str}")
    prefix = match.group(1).upper()
    number = int(match.group(2))
    return prefix, number

def format_serial(prefix, number, padding=3):
    return f"{prefix}{str(number).zfill(padding)}"
# format_serial('UA', 1, 3)    → 'UA001'
# format_serial('UA', 1000, 3) → 'UA1000'

Overlap Detection SQL

-- Deteksi overlap range untuk customer tertentu
SELECT sr.id, sr.prefix, sr.serial_from, sr.serial_to
FROM   seal_serial_range sr
WHERE  sr.customer_id  = %(customer_id)s
  AND  sr.prefix       = %(prefix)s
  AND  sr.serial_from <= %(new_to)s
  AND  sr.serial_to   >= %(new_from)s
  AND  sr.state NOT IN ('cancelled');
